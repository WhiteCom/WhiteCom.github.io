## <span class="text-orange">C++ Undefined Behavior(UB)</span>

[ ABI 란? ]
- ABI 란 Application Binary Interface 의 줄임말로, 컴파일된 바이너리 코드가 서로 상호작용하는 방식을 정의한 하위 레벨의 규약이다. 

비슷한 말이지만 다른 뜻으로 사용되는 API 라는 것도 있다. 
[ API 란? ]
- API 란 코드 작성자가 보는 함수 선언 규칙을 말한다. 

즉, 헤더파일 내의 함수선언이 같아 보여도, 실제로 만들어진 .lib / .dll / .exe 파일 사이의 내부 약속이 다르면 꺠질 수 있다. 

---

## 왜 위험한가? 

예를들어 아래와 같은 것들이 ABI에 포함된다.

1. 함수 인자 전달방식
2. 이름 맹글링 (name mangling)
3. 클래스 메모리 배치 (layout)
4. virtual table 구조
5. Exception 처리 방식
6. RTTI 구조
7. STL 컨테이너 내부 표현
8. struct padding / alignment
9. new / delete 가 어느 CRT 를 쓰는지

위 중 하나라도 다를 경우,
- Link Error
- 런타임 크래시
- 메모리 손상
- dynamic_cast 실패
- delete 사용 시 Heap 충돌
- DLL 경계 넘는 객체 전달 문제
와 같은 문제가 발생할 수 있다. 

---

## 대표적인 위험사례 

<span class="text-orange">1. 컴파일러가 다름</span>

- VS 2008 에서 만든 DLL
- VS 2026 에서 만든 EXE

헤더는 같아도, C++ 클래스 ABI가 다를 수 있다. 

특히 아래의 경우 위험하다. 
- 클래스 객체를 DLL 경계로 직접 주고받기
- std::string, std::vector 전달
- Exception throw / catch across DLL
- 메모리를 DLL에서 할당하고 EXE에서 해제

<span class="text-orange">2. 컴파일 옵션이 다름</span>

예시)
- Debug / Release 혼용
- /MDd, /MD 혼용
- 구조체 정렬 옵션 다름 (#pragma pack, /Zp)
- wchar_t 처리 옵션 차이
- Exception 옵션, RTTI 옵션 차이

위 경우도 ABI가 달라질 수 있다. 

예를들어 struct 크기가 달라질 경우,

```cpp
struct S
{
    char a;
    int b;
};
```

한쪽은 8바이트, 다른쪽은 5바이트처럼 해석될 수 있으며, 
padding 규칙 차이로 실제 메모리 배치(layout)가 달라질 수 있고, 
그 상태로 바이너리 교환할 경우 잘못된 정보를 읽게 된다. 

<span class="text-orange">3. STL을 DLL 경계로 전달</span>

예시)

```cpp
std::string GetName();
void SetData(std::vector<int> v);
```

위와 같은 인터페이스를 DLL export로 공개하면,
컴파일러/ 버전/ CRT 다 다를 떄 깨질 가능성이 매우 높다. 

왜냐하면, std::string, std::vector의 내부구조는 C++ 표준이 '기능'만 보장하지, '메모리 레이어' 까지 고정하지 않기 떄문이다. 

<span class="text-orange">4. 클래스 멤버 변경</span>

라이브러리 배포 후 클래스에 멤버 하나 추가했는데 예전 바이너리와 같이 쓰면 레이아웃이 달라져서 문제 발생 가능성 존재함. 

예시)

```cpp
class A
{
    int x;
};
```

이전 클래스가 위와 같았다고 할 경우,

```cpp
class A
{
    int x;
    int y;
};
```

로 클래스를 바꾸면, 기존 바이너리가 예상한 객체 크기와 달라짐.

멤버변수 추가와 동일하게 virtual 함수 순서변경도 위험하다. 

### API 는 동일해도, ABI 는 꺠질 수 있다는 뜻.

virtual 함수 순서변경을 할 경우 ABI가 깨질 수 있다. 

예시)
```cpp
class Foo
{
public:
    virtual void Do();
    virtual void Func();
    int x;
};
```

이전에 위와 같은 클래스

```cpp
class Foo
{
public:
    virtual void Func();
    virtual void Do();
    int x;
};
```

와 같이 바꿀 경우 소스코드 레벨에선 동일해보임.
그러나, 바이너리에선 두 결과물이 완전히 다르다. 

이는 vtable 의 인덱스 변화가 발생했기 떄문이다. 
컴파일러는 가상함수를 호출할 때, '이름' 으로 찾는 것이 아닌, vtable 내의 인덱스로 접근한다. 

이전 클래스에서 Do 함수는 1번 인덱스, Func 함수는 2번 인덱스였다. 
변경된 클래스에서는 Do 함수는 2번 인덱스, Func 함수는 1번 인덱스이다. 

만약 이런 상태에서 DLL 파일은 예전 상태, EXE 파일은 변경된 상태로 쓴다면, 서로 vtable 의 0번 인덱스에 해당하는 함수호출이 다른 함수를 호출하기 떄문에 문제가 발생한다.

## 언제 특히 조심해야 하는가?

다음 상황이면 ABI 위험성을 크게 의식해야한다. 

1. DLL / SO (shared object - Unix 계열 OS 에서 사용하는 동적 Lib) / shared library 배포
2. 외부 라이브러리 연동
3. 서로 다른 Visual Studio 버전 혼용
4. 구형 프로젝트 마이그레이션
5. 플러그인 구조
6. 엔진/ 툴/ 게임 실행파일이 서로 다른 Build chain 사용
7. Cpp 객체를 모듈 경계 밖으로 노출
